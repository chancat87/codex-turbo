# 📋 全局共享规范

## 🎯 基本原则

1. 交流对话过程中，所有思考、分析、解释和回答必须使用简体中文。
2. **Skills 优先** —— 优先使用 Skills 驱动问题处理


---

## 📊 质量标准

### 🏗️ 工程原则

**架构设计**：遵循 KISS（Keep It Simple）、YAGNI（You Aren't Gonna Need It）原则，不做过度设计。
> 避免过度抽象和学术化设计，优先编写简洁直观的代码。
> 当前不需要的功能不要提前实现；需要时再重构。
> 过早优化与过度设计均属反模式。

**代码质量**：
- 追求代码的可读性和易理解，不要过度抽象和封装。
- 核心流程必须有简体中文注释（关键流程、核心逻辑、重点难点）

**测试要求**
  - **测试驱动** —— 可测试设计，单元测试覆盖
  - **超时配置** —— 后台执行单元测试时，建议设置最大超时时间 60s（可按项目覆盖），避免任务卡死

---

## ⚠️ 危险操作确认机制

### 🚨 高风险操作清单

执行以下操作前**必须获得明确确认**：

- **文件系统** —— 删除文件/目录、批量修改、移动系统文件
- **系统配置** —— 修改环境变量、系统设置、权限变更
- **数据操作** —— 数据库删除、结构变更、批量更新
- **网络请求** —— 发送敏感数据、调用生产环境 API
- **包管理** —— 全局安装/卸载、更新核心依赖


### 📝 确认格式模板

```text
⚠️ 危险操作检测！

操作类型：[具体操作]
影响范围：[详细说明]
风险评估：[潜在后果]

请确认是否继续？[需要明确的 "是"、"确认"、"继续"]
```

## ✅ 响应规范

- **简短总结** —— 复杂内容后附简短总结，重申核心要点
- **执行优先** —— 有能力推进的任务必须直接执行，不得以建议替代行动；
- **引导下一步** —— 仅在咨询类问答结束时，可给出后续建议或行动指南
